# V4.1 项目删除 — 设计文档

## 1. 概述

本文档描述"项目删除"功能的完整设计，重点覆盖并发安全、分布式锁、竞态条件和边界情况的处理。

Stop 机制的完整设计见 `v4-run-lifecycle-design.md`。

---

## 2. 系统架构背景

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  Next.js    │     │   BullMQ     │     │   Worker    │
│  API Routes │────▶│   Queue      │────▶│  Process    │
│  (Web 端)   │     │  (Redis)     │     │  (独立进程) │
└─────────────┘     └──────────────┘     └─────────────┘
       │                                        │
       │            ┌──────────────┐            │
       └───────────▶│  PostgreSQL  │◀───────────┘
                    │  (Prisma)    │
                    └──────────────┘
                           │
                    ┌──────────────┐
                    │    Redis     │
                    │  (锁/SSE)   │
                    └──────────────┘
```

关键约束：
- Web 端（API Routes）和 Worker 是独立进程，无法直接通信
- 协调通道：Redis（分布式锁、SSE pub/sub）和 PostgreSQL（ProjectRun 状态）
- Agent Loop 单次运行可能超过 10 分钟（50 步 × LLM 调用 + shell 命令）
- 写权限由 run-fencing（`assertRunWritable` + `finalizeRun`）保证，不使用 Redis cancel flag

## 3. 分布式锁设计

### 3.1 锁的职责

`project-lock:{projectId}` 保证同一项目同一时间只有一个操作在执行（Worker 任务或 DELETE 操作）。

### 3.2 实现机制

| 特性 | 实现 |
|------|------|
| 互斥获取 | `SET key token EX ttl NX` — 原子操作 |
| Owner token | 随机 UUID，防止误释放 |
| 安全释放 | Lua compare-and-delete：token 匹配才删除 |
| 心跳续租 | `setInterval` 每 TTL/3 执行一次 Lua compare-and-expire |
| 锁丢失保护 | 续租失败时 clearInterval，依赖 run-fencing 兜底 |

### 3.3 Lua 脚本

```lua
-- 释放锁（compare-and-delete）
if redis.call("get", KEYS[1]) == ARGV[1] then
  return redis.call("del", KEYS[1])
else
  return 0
end

-- 续租（compare-and-expire）
if redis.call("get", KEYS[1]) == ARGV[1] then
  return redis.call("expire", KEYS[1], ARGV[2])
else
  return 0
end
```

### 3.4 为什么需要 owner token

没有 token 的锁存在经典问题：

```
T1: Worker A 获取锁（TTL=600s）
T2: Worker A 因 GC/网络暂停，锁过期
T3: DELETE 获取锁（新锁）
T4: Worker A 恢复，调用 redis.del(key) → 删掉了 DELETE 的锁
T5: Worker B 获取锁 → 互斥被破坏
```

有 token 后，T4 的 `del` 变成 compare-and-delete，token 不匹配，操作无效。

### 3.5 为什么需要心跳续租

Agent Loop `maxSteps=50`，单步可能包含：
- LLM 调用：5-30 秒
- Shell 命令（npm install/build）：最多 120 秒
- 文件写入 + 沙箱操作：数秒

总耗时可能超过 10 分钟（默认 TTL 600s）。没有续租，锁过期后 DELETE 可以拿到锁并删除项目，而 Worker 仍在运行。

续租策略：
- 间隔：TTL/3 = 200 秒
- 最坏情况：续租刚执行完，下次续租前锁还剩 TTL - TTL/3 = 400 秒
- 安全裕度：即使一次续租失败（网络抖动），还有 400 秒缓冲

### 3.6 锁丢失时的行为

```typescript
if (!renewed) {
  clearInterval(renewInterval);
  // 锁已丢失，但正确性由 run-fencing 保证：
  // - assertRunWritable 在每个检查点校验 run.status
  // - finalizeRun 使用条件更新，不会覆盖已被 Stop/Delete 修改的状态
}
```

续租返回 false 意味着锁已被其他人获取。此时 Worker 可能继续执行，但：
- 如果 Delete 已经删除了项目，Worker 的 DB 写入会报错，触发 catch → `finalizeRun` 兜底
- 如果 Stop 已经改了 run.status，下一个 `assertRunWritable` 检查点会 throw

锁是优化手段（减少无效执行），不是正确性保证。正确性由 run-fencing 的条件更新保证。

## 4. 项目删除（DELETE /api/projects/:id）

### 4.1 设计原则

- Delete 判断能否删除必须基于 `ProjectRun` 状态，不能只看 `Project.status`
- 不使用 Redis cancel flag（已废弃），写权限由 run-fencing 保证
- 不在获取锁之前设置任何标记（如 `deleting=true`），避免"检查失败但标记已设"导致项目卡死
- 存在 active run 时一律返回 409，不做 force delete

### 4.2 完整流程

```
1. 查询项目（带 userId 权限校验 + include sandboxSession）
2. 项目不存在 → 404
3. 查询 active run（status in [queued, running, cancelling]）
4. 如果存在 active run → 409 "请先停止并等待完成"
5. acquireProjectLock(id) → 获取失败 → 409 "正在处理中"
6. try:
   a. 锁内 double-check active run（防止 step 3 和 step 5 之间有新 run 创建）
   b. 如果仍有 active run → 409
   c. 移除队列中同 projectId 的 waiting/delayed jobs
   d. best-effort kill sandbox（Sandbox.connect → kill）
   e. prisma.project.delete()（级联删除所有子表）
7. finally:
   a. releaseProjectLock(id, token)
8. 返回 200 { success: true }
```

### 4.3 四层防护

```
┌─────────────────────────────────────────────────────┐
│ 第一层：ProjectRun 状态检查                           │
│ 存在 active run（queued/running/cancelling）→ 409    │
├─────────────────────────────────────────────────────┤
│ 第二层：分布式锁                                      │
│ Worker 持有锁 → acquireProjectLock 失败 → 409 拒绝   │
├─────────────────────────────────────────────────────┤
│ 第三层：锁内 double-check                            │
│ 获取锁后再次检查 active run → 捕获竞态窗口            │
├─────────────────────────────────────────────────────┤
│ 第四层：Worker 防御性检查                             │
│ Job 启动时 findUnique → 项目不存在 → 静默退出         │
└─────────────────────────────────────────────────────┘
```

### 4.4 实现代码

```typescript
export async function DELETE(
  _request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params;

  const project = await prisma.project.findUnique({
    where: { id, userId: DEMO_USER_ID },
    include: { sandboxSession: true },
  });

  if (!project) {
    return Response.json({ error: "Project not found" }, { status: 404 });
  }

  // 基于 active run 判断，而非 project.status
  const activeRun = await prisma.projectRun.findFirst({
    where: {
      projectId: id,
      status: { in: ["queued", "running", "cancelling"] },
    },
  });

  if (activeRun) {
    return Response.json(
      { error: "项目有正在执行的任务，请先停止并等待完成" },
      { status: 409 }
    );
  }

  // 获取项目锁，防止与 Worker 竞态
  const { acquired, token } = await acquireProjectLock(id);
  if (!acquired) {
    return Response.json(
      { error: "项目正在处理中，请稍后再试" },
      { status: 409 }
    );
  }

  try {
    // 锁内 double-check：防止获取锁期间有新 run 创建
    const activeRunInLock = await prisma.projectRun.findFirst({
      where: {
        projectId: id,
        status: { in: ["queued", "running", "cancelling"] },
      },
    });

    if (activeRunInLock) {
      return Response.json(
        { error: "项目有正在执行的任务，请先停止并等待完成" },
        { status: 409 }
      );
    }

    // 移除队列中同 projectId 的待执行任务
    const waitingJobs = await agentQueue.getJobs(["waiting", "delayed"]);
    for (const job of waitingJobs) {
      if (job.data.projectId === id) {
        await job.remove();
      }
    }

    // best-effort 关闭沙箱
    if (project.sandboxSession?.sandboxId) {
      try {
        const sandbox = await Sandbox.connect(project.sandboxSession.sandboxId);
        await sandbox.kill();
      } catch { /* ignore */ }
    }

    // 级联删除项目及所有关联记录
    await prisma.project.delete({ where: { id } });
  } finally {
    await releaseProjectLock(id, token);
  }

  return Response.json({ success: true });
}
```

### 4.5 数据库级联删除

Prisma schema 中所有子表配置 `onDelete: Cascade`：

```
Project (delete)
  ├── Message            (cascade)
  ├── ProjectFile        (cascade)
  ├── BuildLog           (cascade)
  ├── AgentRun           (cascade)
  ├── ProjectRun         (cascade)
  ├── SandboxSession     (cascade)
  └── AgentConversation  (cascade)
```

一次 `prisma.project.delete()` 由 PostgreSQL 自动清理所有关联数据。

### 4.6 权限校验

```typescript
where: { id, userId: DEMO_USER_ID }
```

即使当前是 demo 用户，也按 userId 限定查询。未来接入真实认证后，只需替换 `DEMO_USER_ID` 为当前用户 ID。

### 4.7 前端交互

两个入口：
- **侧边栏**：hover 项目时右侧显示 Trash2 图标，点击 confirm 后删除；如果删除的是当前查看的项目，跳转首页
- **首页卡片**：hover 时右上角显示 Trash2 图标，同样带确认

409 响应处理：`alert(data.error)` 提示用户先停止项目。

## 5. 并发问题与竞态分析

### 5.1 TOCTOU：active run 检查与删除之间的竞态

**问题：**
```
T1: DELETE 检查 active run → 没有
T2: Worker 开始消费 job，run.status queued → running
T3: DELETE 执行 prisma.project.delete()
```

**解决：** DELETE 在 active run 检查后还要获取项目锁。Worker 通过 `withProjectLock` 持有锁，DELETE 的 `acquireProjectLock` 会失败，返回 409。

**反向竞态：**
```
T1: DELETE 获取锁
T2: Worker 尝试 withProjectLock → 失败 → job 抛错 → finalizeRun 兜底
```

这是安全的。Worker job 失败后 `finalizeRun` 将 run 标记为 failed。BullMQ 重试时 `findUnique` 发现项目已删除，直接跳过。

### 5.2 Stop 后立即 Delete 的竞态

**问题：**
```
T1: 用户点停止 → run.status = cancelling, project.status = stopped
T2: 用户立即点删除 → active run 检查发现 cancelling → 409
```

**结果：** 安全。Delete 会被 active run 检查拦截（cancelling 属于 active 状态）。用户需要等 Worker 在检查点退出并 finalize 为 cancelled 后，才能删除。

**如果 Worker 已经退出：**
```
T1: 用户点停止 → run.status = cancelling
T2: Worker 检查点感知 → finalizeRun → run.status = cancelled
T3: 用户点删除 → active run 检查通过 → 获取锁 → 删除成功
```

### 5.3 创建 Run 与 Delete 的 TOCTOU

```
T1: DELETE 检查 active run → 没有
T2: 新消息到达 → 创建 run(queued) → 入队
T3: DELETE 获取锁 → 锁内再次检查 → 发现新 run → 返回 409
```

锁内 double-check 能捕获大部分情况。但如果创建 run 的 API 不获取 project lock，仍有极小窗口：

```
T1: DELETE 获取锁 → 锁内检查 active run → 没有
T2: 创建 run API（不需要锁）→ 创建 run → 入队
T3: DELETE 执行 delete → 新 run 被级联删除
T4: Worker 消费 job → findUnique → 项目不存在 → return
```

第一版兜底：Worker 启动时检查项目和 run 是否存在。如果不存在，静默退出，不重试。这个极端场景的后果是"用户刚发的消息被吞"，但概率极低且用户可以重新发送。

如果未来需要更强保证，可以让创建 run 的 API 也获取 project lock，或引入 `project.deletedAt` 软删除。

### 5.4 队列 Job 与删除的竞态

**问题：**
```
T1: 用户创建项目 → enqueueRun → run.status = queued
T2: DELETE 通过检查 + 获取锁 + 删除项目
T3: Worker 消费 job → findUnique → 项目不存在 → return
```

**解决：** 双重防护：
1. DELETE 在锁保护下移除 waiting/delayed jobs
2. Worker 启动时 `findUnique` 检查项目是否存在

### 5.5 锁过期导致的互斥丢失

**问题：**
```
T1: Worker 获取锁（TTL=600s）
T2: 任务执行超过 600s，锁过期
T3: DELETE 获取新锁，删除项目
T4: Worker 继续执行，写已删除项目 → DB 报错
```

**解决：** 心跳续租（每 200s）+ run-fencing 兜底：
- 正常情况：续租保持锁有效，DELETE 拿不到锁
- 异常情况（续租失败）：Worker 后续 DB 操作报错 → catch → `finalizeRun` 兜底

**残余风险：** 如果 Redis 完全不可达超过 400s（续租失败 + 剩余 TTL 耗尽），锁过期。但 Worker 的 DB 写入会因为项目已删除而报错，触发异常处理流程。不会导致数据不一致。

### 5.6 并发删除同一项目

**问题：** 两个 DELETE 请求同时到达。

**解决：** `acquireProjectLock` 是 `SET NX`，只有一个能成功。另一个返回 409。

## 6. 用户操作路径

### 6.1 停止并删除活跃项目

```
用户在 ChatPanel 点击停止按钮（红色 Square 图标）
  → POST /api/projects/:id/stop
  → run.status = cancelling, project.status = stopped
  → Worker 在检查点感知 → finalizeRun → run.status = cancelled
  → 前端 SSE 收到 status_change: stopped

用户在侧边栏 hover 项目，点击删除图标
  → confirm("确定要删除？")
  → DELETE /api/projects/:id
  → active run 检查通过 → 锁获取成功 → 级联删除 → 200
  → 项目从列表消失
```

### 6.2 删除非活跃项目（created/failed/stopped）

```
用户在首页/侧边栏 hover 项目，点击删除图标
  → confirm("确定要删除？")
  → DELETE /api/projects/:id
  → active run 检查通过 → 锁获取成功 → 级联删除 → 200
```

### 6.3 删除活跃项目（被拒绝）

```
用户尝试删除有 active run 的项目
  → DELETE 返回 409: "项目有正在执行的任务，请先停止并等待完成"
  → 前端 alert 提示
```

### 6.4 删除时 Worker 仍在处理（锁保护）

```
用户停止后立即删除（Worker 还在收尾）
  → DELETE active run 检查发现 cancelling → 409: "请先停止并等待完成"
  → 用户等几秒后重试 → Worker 已 finalize → active run 检查通过 → 删除成功
```

## 9. 已知限制与后续演进

### 9.1 当前限制

| 限制 | 影响 | 缓解措施 |
|------|------|----------|
| 删除活跃项目需要先 Stop 再等待 | 用户体验略繁琐 | 前端可串联 stop + 轮询 + delete |
| Redis 完全不可达时锁失效 | 极端情况下互斥被破坏 | run-fencing 兜底 + Redis 高可用 |
| getJobs 遍历全部 waiting jobs | 队列很大时性能差 | 当前单用户 demo 可接受 |
| 极端 TOCTOU 窗口（创建 run 与 delete 并发） | 新消息被吞 | Worker 防御性检查 + 用户可重发 |

### 9.2 后续可选优化

| 方案 | 适用场景 | 复杂度 |
|------|----------|--------|
| 前端串联 stop → poll → delete | 一键删除活跃项目 | 低 |
| 软删除（deletedAt + 定时清理） | 需要回收站/撤销功能 | 中 |
| BullMQ Job ID 按 projectId 索引 | 大量 job 时快速定位移除 | 低 |
| 创建 run 也获取 project lock | 消除 TOCTOU 窗口 | 中（每次发消息都抢锁） |
| Redis Sentinel / Cluster | 锁的高可用保障 | 基础设施层 |
